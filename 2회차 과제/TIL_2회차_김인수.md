1. 오늘 배운 내용

Java OOP 기초 — 절차 지향 프로그래밍과 객체 지향 프로그래밍의 차이
클래스 분리 → 메서드 분리 → 객체 통합의 리팩토링 흐름
오버로딩(Overloading): 같은 이름, 다른 파라미터
생성자(Constructor): 객체 생성 시 초기값 주입
OOP 4대 특성 개요: 캡슐화, 상속, 추상화, 다형성
캡슐화(Encapsulation): 접근 제어자 private으로 필드를 숨기고 메서드로만 접근
this 키워드: 클래스 필드와 메서드 파라미터 이름이 같을 때 구분하는 방법


2. 핵심 정리 (내 언어로)

절차 지향 vs 객체 지향: 절차 지향은 "어떻게 할까(흐름)" 중심이고 객체 지향은 "무엇이 할까(객체)" 중심이다. 음악 플레이어 예시로 보면, 절차 지향은 isOn = true처럼 상태를 직접 건드리는 코드를 순서대로 쭉 쓰는 방식이고, 객체 지향은 player.on()처럼 객체에게 행동을 시키는 방식이다.
절차 지향의 문제: 기능이 추가될수록 변수와 로직이 main 안에 계속 쌓인다. 플레이어가 여러 개면 volume1, volume2... 처럼 변수도 N배로 늘어난다.
캡슐화의 핵심: private으로 필드를 막아두면 외부에서 player.volume = -999 같은 이상한 값을 직접 때려 넣는 걸 컴파일 단계에서 막을 수 있다. 상태 변경은 반드시 검증 로직을 넣을 수 있는 메서드를 통해서만 허용.
this: setVolume(int volume)처럼 파라미터 이름이 필드명과 같을 때 this.volume은 객체의 필드, volume은 전달받은 파라미터를 가리킨다.
